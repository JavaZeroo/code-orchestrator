# UI 信息架构重构：任务中心

版本：2026-07-07
状态：grill 定稿（8 问），三步分期实现
关系：解决「侧边栏『需求』『工作流』两个 tab 看不懂」的问题；是 design-v2「用户只提任务、不挑机器」在 UI 层的落地。

---

## 0. 一句话

把「任务」立为 UI 唯一主语：需求命中与手动 run 统一进 **work_items 血缘树**（已建成但一直没人消费的投影，`api.ts:242`），触发器/流程模板降级为**项目配置**，看板瘦身为**注意力页**，新建任务**对话优先**。侧边栏从 5 个 tab（看板/项目/需求/工作流/会话）变为 4 个：**看板 / 任务 / 项目 / 会话**。

## 1. 现状诊断

- 「需求」tab 实为触发器配置页：主体是 8 字段的新建表单，名不符实。
- 「工作流」tab 混装定义（模板）、启动表单、运行记录三种东西；run 同时出现在看板，用户分不清两处区别。
- 服务端有专为「看得懂」而建的 work_items CQRS 投影（血缘：项目→需求→run→节点→PR/门审批），API 与客户端函数俱全，但**没有任何页面消费**。
- 用户要面对 7 个名词：触发器、需求、工作流定义、运行、会话、审批、项目。

## 2. 八条决议

| # | 决议 | 关键取舍 |
|---|---|---|
| Q1 | **心智模型 = 任务中心**，work_items 树为主视图 | 触发器/模板是管道，藏进项目配置；侧边栏 4 tab |
| Q2 | **任务树不含独立会话** | 树 = 可追溯的编排工作（需求 + 手动 run）；聊天归会话 tab；task_queue 落地后排队任务再进树 |
| Q3 | **看板 = 注意力页**：等我处理 + 进行中 | 删「最近完成」栏；run 级行点击跳任务树；会话级分流（工具审批/进行中 agent）仍归看板——恰是任务树刻意不装的 |
| Q4 | **新建任务对话优先** | 描述要干什么，而不是先挑模板 |
| Q5 | **受理范围 = 选模板 + 现场编排** | agent 双工具：emit_task_plan（选模板填变量）+ 现有 emit_workflow（没合适模板就现场画图，可存为模板）；Designer 不再是独立入口，被「新建任务」对话吸收——全系统只剩一个对话入口 |
| Q6 | **计划卡片确认后启动 + 保留快捷路径** | agent 只出计划，人点「启动」（与 autonomy=manual 哲学一致）；「新建任务」下拉保留「直接选模板」复用现有 StartForm |
| Q7 | **触发器 + intake 日志 → 项目详情页「自动化」区** | 新建项目详情页（概览/自动化/流程模板，v2 预留环境与资源）；seeded 基线、触发失败是配置域诊断信息，不进任务树；任务树只显示真正触发成 run 的需求 |
| Q8 | **三步分期落地** | 每步可用可验收，dogfooding 不断流 |

## 3. 实现拍板（不占决策带宽的部分）

- **任务树按项目过滤**：`work_items` 加 `projectId` 列（additive，手写 SQL，守迁移纪律）；投影器在 `run.started` 时从 runs 表回查 projectId（事件 payload 顺手补上，现在只有 `{defId,name,vars}`）；`rebuildWorkItems()` 重放回填存量。投影中 `project` 类型根节点不再作为 UI 树根（项目已由全局切换器选定）。
- **树的实时性**：复用现有事件流，对 `run.*` / `requirement.*` / `approval.*` 失效刷新。
- **树节点点击**：run → 现有 RunView 内嵌任务 tab；节点/会话 → 会话视图；PR → 外链。
- **通知中心 openRunId 深链**改跳任务 tab 的 RunView。
- **受理 agent 机制**：复制 designer 套路——spawn 带专用标志的会话 + 注入工具 + 事件回流 UI 出计划卡片。

## 4. 三步分期

- **① 服务端 projectId 贯通**：work_items 加列 + 投影器补链 + `/api/work` 项目过滤 + 重放回填。纯 additive，旧 UI 照常跑。
- **② 前端 IA 重摆**：新增任务 tab（血缘树 + RunView 内嵌）；新建项目详情页（自动化 = 触发器配置 + intake 日志搬家；流程模板 = 定义列表搬家）；看板瘦身；删 TriggersPage / WorkflowsPage 两个 tab。此阶段「新建任务」先用「直接选模板」表单顶上。
- **③ 对话受理**：emit_task_plan 工具 + 计划卡片 + 确认启动，补上「新建任务」的默认入口；Designer 入口合并进来。

## 5. 遗留细节（实现时再定）

- 已有模板的对话式**编辑**（designer 目前只会新建）。
- 任务树的过滤器形态（状态筛选、只看运行中等）。
- 「需求」一词保留为树中 issue 来源任务的类型标签，不再作为 tab 名。
