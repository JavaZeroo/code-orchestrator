# 对话优先重构（conversation-first）

2026-07-08 复盘 grill 产物。起因：owner 反馈「做出来的网站不是我想要的东西，差距有点大」。
诊断结论：**病根不是单个页面，而是名词体系**——看板/任务/会话/工作流/门禁是按系统实现分的类，
不是按 owner 的心智模型分的类。owner 的原生模型是**多线程对话**（像 Claude/ChatGPT 的会话列表）：
对话是唯一容器，项目是对话的背景，其余一切都是对话的属性或内容。

本文档取代 `design-ui-ia.md` 的导航结构部分（任务中心方案）；其事件溯源、受理链路、
门禁机制等后端设计全部沿用。

## 六条决议

1. **心智模型 = 多线程对话**。对话是唯一容器；词汇统一为「对话」——
   会话（session）/任务（work item）/运行（run）三套 UI 名词合一，均以对话呈现。
2. **一个活 = 一条对话**。流水线跑活（PM 规划→开发→CI→评审→合并门禁）从头到尾就是
   一条对话：规划是消息、开发 agent 的进展是可折叠的活动卡片、PR/CI 是状态卡、
   审批是带按钮的卡片；owner 随时可以在线程里插话改需求。没有第二个地方要看。
3. **导航 = 项目切换器 + 新对话 + 当前项目的线程列表**。
   等 owner 处理的线程自动置顶+红点（看板的职能被吸收进排序），其余按最近活跃排。
   看板、任务两个 tab 拆除（第①期），项目 tab 拆除（第③期）。
4. **项目 = 干净的工作区**（保留左上角全局切换器，类似 Slack 切 workspace）。
   跨项目注意力补洞：切换器下拉条目上标红点+待办数；右上铃铛是跨项目聚合流，
   点一条直接跳到那条线程（自动切项目+选中）。
5. **「工作流」作为名词从导航消失**，变成项目设置里的「自动化」栏：
   什么事件自动触发什么流程（issue → dogfood 流水线）。React Flow 编辑器降级为
   设置里的二级精调工具。远期演化方向：对话里说一句话就能改自动化（本期不做）。
6. **机器、forge 绑定、通知偏好 → 设置**；**血缘 = 对话内的卡片互链**
   （返工轮、衍生对话互相跳转），独立血缘树页面取消。

## 三期实施（每期一个 [dogfood] issue，期末截图验收、网站可用）

### 第①期 导航收敛
- 线程列表上位为首页：现「会话」页成为 app 主体，默认路由。
- 线程列表数据源融合：sessions + workflow runs（当前项目）混排为统一线程项；
  run 线程项本期点开仍进现有运行视图（详情融合放第②期），保证可达性不倒退。
- 等我处理置顶：waiting_human 的线程置顶 + 红点徽章。
- 项目切换器条目红点 + 各项目待办数；右上铃铛聚合跨项目待办，点击跳线程。
- 拆除看板、任务两个 tab。项目 tab 本期保留。
- 纯前端 + 允许 additive 聚合 API（如按项目 waiting 计数）；不动 schema、不加迁移。

### 第②期 run→对话融合（工程量最大）
- 一条流水线 run 的全部内容并入同一条对话时间线：各节点 agent 会话消息合流、
  节点状态迁移为状态卡、PR/CI 为状态卡、门禁审批为带按钮的卡片（点了即 decideGate）。
- 开发 agent 的过程消息默认折叠成活动卡（进行中显示实时状态，可展开全细节）。
- owner 在线程里发消息 = 给当前活跃节点的会话插话（nudge 既有机制）。
- 事件溯源已含全部素材（events 表），主要工作是聚合视图 API/WS + 前端渲染。

### 第③期 项目页拆除
- 项目设置成型：基本信息、仓库/镜像、成员凭据、「自动化」栏
  （触发规则 + 二级入口进 React Flow 编辑器）。
- 「工作流」「任务」名词从 UI 全部消失；血缘以对话内互链卡片呈现。
- 机器管理确认收进设置。

## 验收基线

- 每期合并部署后用 headless chrome 截图验收（流水线对像素盲，人工/主会话补位）。
- 每期结束网站完整可用，无功能孤岛（可达性不倒退是硬条件）。
